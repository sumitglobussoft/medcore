# MedCore Mobile E2E — Maestro

End-to-end regression coverage for the patient-facing Expo app.
Powered by [Maestro](https://maestro.dev) — YAML-driven, no native
rebuild, no extra npm dependency.

## Why Maestro (and not Detox / Appium)

- **Zero native footprint.** Maestro drives the already-installed app
  over adb / `xcrun simctl`. No `eas build --profile test`, no
  `detox test --configuration ios.sim.debug`, no keeping a separate
  release target in sync with production. This is the killer feature
  for an Expo-based app where `apps/mobile/ios/` and
  `apps/mobile/android/` are regenerated by prebuild and we don't
  check them in.
- **No devDep bloat.** Maestro is a CLI binary, not a node module. It
  installs to `~/.maestro/bin/maestro` and is invoked directly by the
  shell script in this folder. `apps/mobile/package.json` does NOT
  list it — adding it there would be a lie.
- **Lowest author cost.** The flow files here are 20-50 lines of
  YAML. A Detox equivalent would need a `jest-circus` test, custom
  matchers, and `device.launchApp(...)` state management per-test.

## Prerequisites

1. **Install Maestro** once per dev machine:
   ```bash
   curl -Ls "https://get.maestro.mobile.dev" | bash
   export PATH="$HOME/.maestro/bin:$PATH"      # add to .bashrc / .zshrc
   maestro --version                            # verify
   ```
   Windows users: install via the Maestro Windows instructions at
   <https://maestro.dev/docs/getting-started/installing-maestro>
   and run these flows under WSL2 (the shell script assumes bash) or
   invoke `maestro test ...` directly from PowerShell.

2. **A running device.** One of:
   - Android emulator booted via Android Studio (preferred for CI).
     `adb devices` should show an `emulator-5554` row.
   - iOS Simulator booted via Xcode. `xcrun simctl list devices`
     should show `(Booted)` for your target.
   - A physical phone with USB debugging on, connected via cable.

3. **The app installed on that device.** Easiest path: run the Expo
   dev server once and open via the Expo Go client or via `npm run
   android` / `npm run ios`:
   ```bash
   npm --prefix apps/mobile run dev            # starts Metro
   npm --prefix apps/mobile run android        # installs + launches once
   ```
   After that first install the app is resident on the device and
   Maestro can drive it without the dev server running (Metro is only
   needed if the JS bundle hasn't been baked in; for EAS dev or
   release builds it's optional).

4. **Seeded patient account.** The flows assume
   `patient1@medcore.local` / `patient123` exists in the target
   backend. For local runs, seed via:
   ```bash
   npm --prefix apps/api run seed:demo
   ```
   For CI / staging, export `PATIENT_EMAIL` and `PATIENT_PASSWORD` at
   invocation time.

## Running the flows

From the monorepo root:

```bash
# All flows, default device, default credentials.
apps/mobile/e2e/run.sh

# Single flow.
apps/mobile/e2e/run.sh login.yaml

# Against staging.
PATIENT_EMAIL=qa.patient@staging.medcore.local \
PATIENT_PASSWORD=stagingpw \
apps/mobile/e2e/run.sh

# Pin a specific device.
MAESTRO_DEVICE=emulator-5554 apps/mobile/e2e/run.sh
```

Artifacts (per-flow stdout/stderr + JUnit XML) land under
`apps/mobile/e2e/artifacts/`. The runner exits non-zero if any flow
fails, so it composes cleanly with CI.

## The flow inventory

| File | Scope |
|---|---|
| [`login.yaml`](login.yaml) | Launch app → login → land on patient dashboard |
| [`view-prescriptions.yaml`](view-prescriptions.yaml) | Rx list renders; tap the first card to expand detail |
| [`view-bill.yaml`](view-bill.yaml) | Billing tab loads and at least one invoice row renders |
| [`view-lab-results.yaml`](view-lab-results.yaml) | Lab Explanations screen mounts, accepts a lab order id |
| [`book-appointment.yaml`](book-appointment.yaml) | AI-assisted booking funnel: triage chat round-trip |
| [`ai-triage-chat.yaml`](ai-triage-chat.yaml) | Red-flag symptom trips the emergency banner |
| [`adherence-mark-dose.yaml`](adherence-mark-dose.yaml) | Tap "Dose at 08:00" and assert the chip flips to taken |

Every flow starts from a clean app state (runner calls `maestro stop`
before each) and re-runs the login flow via `runFlow:` so they're
independent and parallelisable on Maestro Cloud.

## CI strategy

### Option A — Maestro Cloud (recommended)

Maestro Cloud runs flows on real iOS / Android devices in the
background. The upload is a single command, and the service returns a
signed URL with per-flow video + screenshot timeline.

```bash
# Once, per CI job:
maestro cloud \
  --app-file <path/to/patient-build.apk> \
  --api-key "$MAESTRO_CLOUD_API_KEY" \
  apps/mobile/e2e/
```

Costs scale with device-minutes. Given our seven-flow suite runs in
~3 minutes wall time, this is ~$1/run on their current plans.
Secrets live as CI environment variables — never commit the API key.

### Option B — Self-hosted (follow-up; not required for v1)

Alternative path if we want to avoid Maestro Cloud costs:

1. Bake an Android emulator + Maestro into a Docker image
   (`reactnativecommunity/react-native-android` + the Maestro install
   curl as a `RUN` step).
2. Mount the signed APK produced by `eas build --profile
   preview --platform android --non-interactive` into the container.
3. Run `maestro test /e2e` with `-Djava.awt.headless=true`.

This is doable but heavy — a headless Android emulator eats ~4 GB
RAM and each CI job's cold start is 90+ seconds while the AVD snapshot
loads. Defer until a cost/benefit analysis against Maestro Cloud
justifies it.

### Option C — Local-only regression gate

Simplest option: run `apps/mobile/e2e/run.sh` on a developer laptop
before every release-candidate build, and require green flows before
cutting the build. Zero CI cost, zero infra, but relies on human
discipline. Use this until Maestro Cloud / self-hosted is wired up.

## Troubleshooting

**`error: maestro is not installed`** — install it, then add
`$HOME/.maestro/bin` to PATH.

**`no devices online`** — Maestro can't find a running
emulator/simulator. Boot one via Android Studio or Xcode, confirm with
`adb devices` / `xcrun simctl list devices`, then retry.

**Flow times out on the first `assertVisible`** — the app probably
isn't installed on the target device, or it's been installed with a
different bundle id. Confirm `maestro -a com.medcore.app start`
launches the right app. Override via `MAESTRO_APP_ID` if running a
preview build with a different id.

**Login flow fails with "Invalid credentials"** — the API likely
isn't seeded with `patient1@medcore.local`. See the seed step in
Prerequisites, or pass real creds via `PATIENT_EMAIL` /
`PATIENT_PASSWORD`.

**Ionicon glyph is matched as text** — some flows rely on visible
labels (e.g. "Rx" or "Bills"). If a future refactor replaces a label
with only an icon, the flow will need `id:` (testID) selectors. Add
`testID="..."` to the component and point the flow at it.

## Where this sits in the test pyramid

- **Unit / render tests** (`apps/mobile/__tests__/*.test.tsx`, 39
  suites, sub-second each) run on every commit via jest. See
  [`../TESTING.md`](../TESTING.md).
- **E2E flows** (this folder) run pre-release / nightly / pre-merge
  on a device. Minutes each, not seconds. Not intended for the fast
  local jest loop.
- **API integration tests** (`apps/api/src/test/`) cover the backend
  contracts that these flows depend on. If an E2E flow fails and the
  API tests are green, the bug is in the mobile client (or in the
  flow itself).
