# Proposed Prisma models — Ops forecasting features

The ops-forecasting features (PRD §7.3) introduce one new persistence concern:
reviewable AI staff-roster proposals. Because this PR is forbidden from
mutating `packages/db/prisma/schema.prisma`, the model is enumerated here so
the schema owner can drop it in during the next migration window.

Until the model lands, `apps/api/src/routes/ai-roster.ts` persists proposals
to a JSON file under `apps/api/data/ai-roster/proposals.json` (path override
via `AI_ROSTER_STORE_PATH`). All three endpoints read/write through a thin
store wrapper so swapping to Prisma is a one-file change.

## `StaffRosterProposal`

```prisma
enum RosterProposalStatus {
  PROPOSED
  APPLIED
  REJECTED
}

model StaffRosterProposal {
  id             String               @id @default(uuid())
  status         RosterProposalStatus @default(PROPOSED)
  startDate      DateTime             @db.Date
  days           Int                  // 7 or 14
  department     String
  // Full scheduler output: { proposals: [{date, shifts: [...]}], summary, ... }
  proposal       Json
  // Solver soft warnings surfaced to the admin.
  warnings       Json
  createdBy      String               // User.id of admin who generated the proposal
  createdAt      DateTime             @default(now())
  appliedAt      DateTime?
  appliedBy      String?              // User.id of admin who applied the proposal

  tenantId       String?
  tenant         Tenant?              @relation(fields: [tenantId], references: [id], onDelete: SetNull)

  creator        User                 @relation("RosterProposalCreator", fields: [createdBy], references: [id])
  applier        User?                @relation("RosterProposalApplier", fields: [appliedBy], references: [id])

  @@index([status])
  @@index([createdAt])
  @@index([department, startDate])
  @@index([tenantId])
  @@map("staff_roster_proposals")
}
```

### Companion `User` relations

```prisma
model User {
  // … existing fields …
  proposalsCreated StaffRosterProposal[] @relation("RosterProposalCreator")
  proposalsApplied StaffRosterProposal[] @relation("RosterProposalApplier")
}
```

### `tenantScopedPrisma` wiring

Add `"StaffRosterProposal"` to `TENANT_SCOPED_MODELS` in
`apps/api/src/services/tenant-prisma.ts` so reads/writes are automatically
scoped to the current tenant (the route already runs inside
`withTenantContext`).

### Migration plan

1. Add the enum + model to `schema.prisma`.
2. `npx prisma migrate dev --name add_staff_roster_proposal`.
3. In `ai-roster.ts` replace the `store.*` helpers with
   `prisma.staffRosterProposal.*` — the surface matches 1:1
   (`create` / `findUnique({ where: { id } })` / `update({ where: { id }, data })` / `findMany({ orderBy: { createdAt: 'desc' } })`).
4. Add `"StaffRosterProposal"` to `TENANT_SCOPED_MODELS`.
5. Delete `apps/api/data/ai-roster/` — one-time migration of existing rows is
   optional (admin console can re-propose).
