# Proposal — Partial Unique Index: One ACTIVE Admission per Patient

Status: PROPOSED (Issue #37, Apr 2026)
Scope: `admissions` table, `patientId` column, scoped to `status = 'ADMITTED'`.

## Why

MedCore production has observed rows where the same patient (e.g. Rahul
Sharma, Priya Mehta) appears on two beds simultaneously with
`status = 'ADMITTED'`. A patient cannot physically occupy two beds. The
`apps/api/src/routes/admissions.ts` create handler now rejects a second
ACTIVE admission with HTTP 409, but that guard is best-effort only — a
race between two concurrent POSTs, a direct SQL insert, or a future code
path bypassing the handler would reintroduce the bug. The authoritative
fix lives at the database: a partial unique index.

## Proposed SQL

```sql
-- Postgres-specific partial unique index.
-- Enforces: for every patientId, at most one admissions row with status='ADMITTED'.
-- DISCHARGED and TRANSFERRED rows are unconstrained (historical admissions
-- for the same patient must coexist).
CREATE UNIQUE INDEX IF NOT EXISTS one_active_admission_per_patient
  ON admissions (patientId)
  WHERE status = 'ADMITTED';
```

## Why this is NOT in `schema.prisma`

Prisma's schema language does **not** express partial unique indexes
natively. `@@unique([patientId])` with a `where` clause is not supported
as of Prisma 5.x. Encoding it in schema.prisma would either:

1. Over-constrain (make `patientId` globally unique across admissions —
   breaks every discharged-then-readmitted patient), or
2. Silently drop the condition (Prisma ignores `where` on `@@unique`).

Follow-up migration must therefore hand-write the SQL. Suggested file:

```
packages/db/prisma/migrations/<TIMESTAMP>_admissions_one_active_per_patient/migration.sql
```

with the exact `CREATE UNIQUE INDEX ...` above.

## Prerequisite: duplicates must be cleaned first

Creating the index on a table that already contains duplicate ACTIVE
admissions will fail with a unique-violation error. Run the cleanup
script first:

```bash
# inspect (safe, zero writes):
npx tsx scripts/dedup-active-admissions.ts

# apply (flips duplicates to DISCHARGED + frees their bed):
npx tsx scripts/dedup-active-admissions.ts --apply
```

Only after the script reports `duplicatesRemaining: 0` should the
migration land.

## Tenant scoping note

Admissions are tenant-scoped via `tenantId`. A cross-tenant collision on
`patientId` shouldn't occur because Patient IDs are already unique
(UUIDs, and `Patient` has its own tenant scoping), so the plain
`(patientId)` partial index is correct. If we ever reuse patient IDs
across tenants we'd need `(tenantId, patientId)` instead — flag for
review when that refactor lands.

## Enum extension (optional, separate decision)

The cleanup script would prefer to mark each duplicate as
`DISCHARGE_AGAINST_MEDICAL_ADVICE` so operators know these rows were
closed by automation, not by clinical discharge. The current
`AdmissionStatus` enum only has `ADMITTED | DISCHARGED | TRANSFERRED`.
Until the enum grows a `DISCHARGED_AGAINST_MEDICAL_ADVICE` variant, the
script sets `status = DISCHARGED`, `conditionAtDischarge = "UNCHANGED"`,
and annotates `dischargeNotes` with
`[Auto-closed: duplicate admission detected, see ticket #37]`. That's
fully reversible (the notes are the forensic trail).

## Rollout

1. Ship the service-layer guard (done, this PR).
2. Run the dedup script in dry-run, review stderr log.
3. Run the dedup script with `--apply` during a quiet window.
4. Verify `SELECT patientId, COUNT(*) FROM admissions WHERE status='ADMITTED' GROUP BY patientId HAVING COUNT(*) > 1` returns zero rows.
5. Land the hand-written migration creating the partial unique index.

Step 5 depends on step 4. Do not reorder.
