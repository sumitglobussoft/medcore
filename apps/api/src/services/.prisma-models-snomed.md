# SNOMED-CT Subset — Proposed Prisma Model

Status: **DESIGN ONLY — no schema changes applied.** This document proposes
the single model required to close the PRD §3.5.4 / §3.7.1 gap around a
deterministic medical-taxonomy anchor for the triage specialty router.
Do **not** edit `packages/db/prisma/schema.prisma` from this file until the
migration PR has been reviewed and signed off by the clinical advisory
board and by the team coordinating the schema-change freeze.

## Why

PRD §3.5.4 calls for mapping captured symptoms to candidate specialties
using a medical taxonomy (SNOMED-CT subset + ICD-10 specialty mappings)
**combined with** an LLM reasoning step. Today the doctor-ranking path is
pure LLM reasoning — there is no deterministic symptom → specialty anchor.
Until this model lands, the service reads the curated JSON file directly
(`packages/db/seed-data/snomed-subset.json`) so the mapping is available
immediately; once the DB row exists and the seed has been run, the service
prefers DB reads (cheaper to query, admin-editable).

## 1. `SnomedConcept` — global reference catalogue (not tenant-scoped)

Follows the same shape as the existing `Icd10Code` model (line 1054 of
`schema.prisma`): a global directory of reference codes, not tenant data.

```prisma
// ─── SNOMED-CT CURATED SUBSET ─────────────────────────────
// Curated subset (~120 concepts) of common OPD presentations used by the
// AI triage router (PRD §3.5.4 / §3.7.1) as a deterministic
// symptom → specialty anchor. Not the full SNOMED release — see the
// licensing note in seed-data/snomed-subset.json.
model SnomedConcept {
  id            String   @id                 // SNOMED conceptId OR LOCAL-xxx
  term          String                       // Fully Specified Name
  synonyms      Json                         // string[] incl. Hindi transliterations
  specialtyTags Json                         // string[] — normalised SCREAMING_SNAKE
  redFlagTerms  Json                         // string[] — consumed by red-flag.ts
  category      String                       // e.g. "cardiac", "neurological"
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@index([category])
  @@index([term])
  @@map("snomed_concepts")
}
```

### Field notes

- `id` — the SNOMED conceptId (numeric string, e.g. `"25064002"`) when
  known; otherwise `LOCAL-<KIND>-<NNN>` for MedCore-internal placeholders
  (hair loss, heat stroke, snakebite, etc. that don't have a precise
  SNOMED hook). Using a string id (not uuid) keeps the natural key intact
  and makes the row directly look-upable from the JSON seed.
- `synonyms` — stored as `Json` (string array). Includes Hindi
  transliterations (Devanagari + Romanised) for every common complaint,
  so the substring matcher in `snomed-mapping.ts` catches
  `"sar dard"` → Headache, `"seene mein dard"` → Chest pain, etc.
- `specialtyTags` — string array of normalised specialty identifiers
  (`GENERAL_MEDICINE`, `CARDIOLOGY`, `NEUROLOGY`, …). Upper-case snake case
  keeps them diff-friendly and trivially comparable to the specialty
  strings emitted by the LLM (which are normalised in the same function).
- `redFlagTerms` — string array consumed by
  `apps/api/src/services/ai/red-flag.ts` as an additional keyword layer
  layered on top of the existing regex patterns. A `"thunderclap"` term
  on the Headache concept surfaces as "Possible subarachnoid haemorrhage"
  in triage replies.
- `category` — coarse grouping (`cardiac`, `respiratory`, …) used for
  admin UI filters and for bucketed evaluation metrics; indexed.

### Not in this model (intentionally)

- `icd10Code` linkage is **not** in this table. ICD-10 already has a
  directory (`Icd10Code` — line 1054) and the PRD lists the two
  taxonomies as independent inputs. A future `SnomedIcdCrosswalk` join
  table can bridge them without widening `SnomedConcept`.
- `tenantId` — explicitly not tenant-scoped. Reference catalogues are
  shared across hospitals, exactly like `Icd10Code` and `Medicine`.

## 2. Migration plan

1. **Migration A (this proposal).** Add `SnomedConcept` table + indexes.
2. **Seed.** Run `npm run db:seed-snomed` (registered in
   `packages/db/package.json`) which reads
   `packages/db/seed-data/snomed-subset.json` and upserts each entry.
   Idempotent — safe to re-run after edits.
3. **Cutover.** `snomed-mapping.ts` reads from DB when available and
   falls back to the JSON file when the table is empty or the query
   throws. No behaviour change for patients during the rollout window.
4. **Future: `SnomedIcdCrosswalk`** (deferred) — a second migration that
   links SNOMED concepts to ICD-10 codes for the Assessment section of
   SOAP notes. Not required for v1 of §3.5.4.

## Do NOT

- Do **not** copy the full SNOMED release into this model. The curated
  subset (~120 concepts) is what the SNOMED International Affiliate
  Licence permits for India-deployed clinical software. Redistributing
  the full release, even internally, would breach the licence. See the
  `_meta` block in `packages/db/seed-data/snomed-subset.json`.
- Do **not** merge the migration without clinical-advisory-board sign-off
  on the specialty tagging. The router's behaviour depends on these tags
  being clinically sensible.
- Do **not** inline this model into `Icd10Code`. They're distinct
  taxonomies that happen to overlap; keeping them as separate tables
  mirrors how every other EHR handles the distinction.
