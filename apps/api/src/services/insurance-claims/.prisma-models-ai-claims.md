# AI Claims — Proposed Prisma Schema Additions

Status: **PROPOSAL ONLY — no schema changes applied.** Captures the schema
deltas that the AI-drafted-claims and denial-risk workflows would benefit
from once a coordinated migration PR is opened.

Until these land, the service code in `ai-coder.ts` + `denial-predictor.ts`
falls back to the `SUBMITTED` enum value plus a `notes` marker
(`[AI DRAFT] Needs review`) so the feature works without schema changes.

---

## 1. Extend `NormalisedClaimStatus`

Add a pre-submission lifecycle stage so reception can confirm an AI-generated
draft before we fire it off to the TPA:

```prisma
enum NormalisedClaimStatus {
  DRAFT_PENDING_REVIEW   // NEW — AI-drafted claim awaiting manual confirmation
  SUBMITTED
  IN_REVIEW
  QUERY_RAISED
  APPROVED
  PARTIALLY_APPROVED
  DENIED
  SETTLED
  CANCELLED
}
```

Notes:
- The existing `SUBMITTED` default on `InsuranceClaim2.status` stays the same.
- Reception UI filter `status=DRAFT_PENDING_REVIEW` surfaces the review queue
  (today it's emulated by `notes LIKE '[AI DRAFT]%'`).

## 2. New fields on `InsuranceClaim2`

```prisma
model InsuranceClaim2 {
  ...existing fields...

  // AI-drafted claim provenance
  aiDraftedFromConsultationId String?     // FK → Consultation.id
  aiDraftReviewedBy            String?    // User.id that confirmed the draft
  aiDraftReviewedAt            DateTime?

  // Denial risk cache — most recent predictor output
  denialRiskLevel   String?     // "low" | "medium" | "high"
  denialRiskReasons Json?       // string[]
  denialRiskAt      DateTime?
}
```

## 3. New model: `ClaimDenialHistory`

Stores per-hospital denial patterns so the Sarvam LLM layer can do
qualitative risk scoring grounded in real rejection history:

```prisma
model ClaimDenialHistory {
  id             String   @id @default(uuid())
  tpaProvider    TpaProvider
  icd10Code      String?
  procedureCode  String?
  denialReason   String
  denialCount    Int      @default(1)
  firstSeenAt    DateTime @default(now())
  lastSeenAt     DateTime @default(now())

  tenantId       String?
  tenant         Tenant?  @relation(fields: [tenantId], references: [id], onDelete: SetNull)

  @@index([tpaProvider, icd10Code])
  @@index([tenantId])
  @@map("claim_denial_history")
}
```

Populated by a nightly job that scans `InsuranceClaim2` rows where
`status = "DENIED"` and aggregates rejection reasons per
`(tpaProvider, icd10Code, procedureCode)`.

## 4. Policy-limits table (optional)

To turn the "amount exceeds policy limit" rule from a warning into a
blocking check we'd also need:

```prisma
model InsurancePolicyLimit {
  id            String      @id @default(uuid())
  tpaProvider   TpaProvider
  insurerName   String
  // Sum-insured / room-rent caps for common plan tiers. Until the hospital
  // loads their rate card, this stays empty and predictor degrades to a
  // heuristic check on amountClaimed vs. Invoice.totalAmount.
  sumInsured        Float?
  roomRentCapPerDay Float?
  maxDailyCharges   Float?

  @@index([tpaProvider, insurerName])
  @@map("insurance_policy_limits")
}
```

## Code paths that would flip after migration

| Today (fallback)                                 | After migration                                 |
| ------------------------------------------------ | ----------------------------------------------- |
| `status: "SUBMITTED"` + `notes: "[AI DRAFT]..."` | `status: "DRAFT_PENDING_REVIEW"`                |
| `GET /pending-drafts` filters on `notes LIKE`    | Filters on `status = DRAFT_PENDING_REVIEW`      |
| Denial predictor re-runs every request          | Reads cached `denialRiskLevel` if fresh (<1hr) |
| LLM layer has no real history                   | LLM layer ingests `ClaimDenialHistory` rows     |
