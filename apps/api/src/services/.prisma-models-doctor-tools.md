# Prisma Model Proposals — Doctor-Facing AI Tools (Apr 2026)

The three doctor-facing AI features (Differential Diagnosis CDS, Smart Follow-up
Scheduler, Lab Result Intelligence) introduce optional per-record AI payloads.
The route handlers implement graceful fallbacks so the features ship without
schema changes, but adding these columns unlocks first-class persistence and
avoids the JSON-in-notes workaround in `ai-lab-intel.ts`.

## LabResult — add AI interpretation columns

```prisma
model LabResult {
  // ... existing fields ...

  // AI Lab Intelligence (Apr 2026) — populated by
  // POST /api/v1/ai/lab-intel/:labResultId/persist
  aiAnalysis     Json?
  aiAnalyzedAt   DateTime?

  // ... existing relations + @@index ...
}
```

Shape of `aiAnalysis`:

```ts
{
  interpretation: string;
  trend: "improving" | "stable" | "worsening" | "unknown";
  baselineComparison: string;
  recommendedActions: string[];
  urgency: "routine" | "soon" | "urgent";
}
```

Until these columns land, `apps/api/src/routes/ai-lab-intel.ts` persists the
analysis by prefixing `LabResult.notes` with a `[AI_INTEL]{...}[/AI_INTEL]`
block. Migrating to proper columns only requires swapping the `update()` payload
in the `/persist` handler.

## AIScribeSession — add precomputed follow-up suggestion

```prisma
model AIScribeSession {
  // ... existing fields ...

  // Smart Follow-up (Apr 2026) — populated by the hook fired after
  // /finalize. See ai-followup.ts.
  followUpSuggestion Json?

  // ... existing relations + @@index ...
}
```

Shape of `followUpSuggestion`:

```ts
{
  consultationId: string;
  suggestedDate: string;   // ISO yyyy-mm-dd
  slotStart: string | null;
  doctorId: string;
  reason: string;
  fallbackUsed: boolean;
}
```

When this column lands, the finalize hook in `ai-scribe.ts` should call
`suggestFollowUp(consultationId)` and write the result to
`followUpSuggestion`; the web follow-up queue page can then list sessions
where `followUpSuggestion IS NOT NULL` directly instead of walking the
consultation list.

## Indices

No new indices required — both payloads are consumed via the primary key.
