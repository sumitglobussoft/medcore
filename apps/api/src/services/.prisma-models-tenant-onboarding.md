# Tenant onboarding — schema follow-ups (deferred)

This file tracks schema changes that WOULD improve the tenant-onboarding
feature but were intentionally NOT made as part of the initial rollout so
we can keep `packages/db/prisma/schema.prisma` untouched during hot-fix
windows. Ticket them when the next schema-churn window opens.

## Current strategy (no schema change)

Per-tenant onboarding state is persisted to the existing `SystemConfig`
table using the key prefix

    tenant:<tenantId>:onboarding_step_<stepName>_completed_at

The value is an ISO-8601 timestamp. `SystemConfig.key` is globally unique
and not tenant-scoped, so the `tenant:<id>:` prefix gives us a per-tenant
keyspace on top of the shared table. See `services/tenant-provisioning.ts`
`tenantConfigKey()` and the `/api/v1/tenants/:id/onboarding` endpoints.

Per-tenant hospital identity (name, phone, gstin, email, address) uses
the same prefix scheme — e.g. `tenant:<id>:hospital_name` — so PDF and
notification renderers can shift to tenant-aware lookups incrementally
without a schema migration.

## Potential future fields (nice-to-have)

If/when a schema window opens, consider adding the following to the
`Tenant` model:

```prisma
model Tenant {
  // existing fields …
  onboardingCompletedAt DateTime?
  primaryContactEmail   String?
  primaryContactPhone   String?
  gstin                 String?
  address               String?
}
```

Or, alternatively, a dedicated `TenantConfig` table:

```prisma
model TenantConfig {
  id        String   @id @default(uuid())
  tenantId  String
  key       String
  value     String
  updatedAt DateTime @updatedAt

  tenant Tenant @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@unique([tenantId, key])
  @@index([tenantId])
  @@map("tenant_config")
}
```

`TenantConfig` would:
1. Make per-tenant config a first-class relation (cascade delete on tenant
   removal).
2. Get rid of the `tenant:<id>:` key-prefix hack.
3. Give us a clean target for the `Tenant.onboardingCompletedAt` rollup.

Migration strategy when the time comes:
1. Create `TenantConfig` table.
2. Backfill rows from `SystemConfig` WHERE key LIKE 'tenant:%:%'.
3. Update `tenantConfigKey()` to route reads/writes to `TenantConfig`.
4. Drop the legacy `SystemConfig` rows in a follow-up migration.

## Why we are not doing it now

- The 2026-04-24 migration train was finalised by hand-crafted migration
  `20260424000002_admission_dama_and_tenant_extension`. Adding another
  table to the same train risks conflicts.
- The SystemConfig prefix gives us the same runtime behaviour with zero
  schema risk.
- The onboarding UI is operator-facing; throughput is tiny, so there is
  no performance pressure on the shared `SystemConfig` table.
