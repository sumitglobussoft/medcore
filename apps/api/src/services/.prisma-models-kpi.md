## 1. `PatientFeedback.appointmentId` (for Â§3.9 CSAT)

**Why.** Today we approximate "CSAT from the AI flow" by finding the patients
who booked through AI triage (the `AI_TRIAGE_SUMMARY` marker embedded in
`appointment.notes`) and averaging any feedback rows those patients submitted
in-window. That's the wrong denominator â€” a patient who also has a walk-in
consultation in the same window will have their unrelated rating folded in.

**Proposed change.**

```prisma
model PatientFeedback {
  // ... existing fields ...
  appointmentId String?
  appointment   Appointment? @relation(fields: [appointmentId], references: [id])
  @@index([appointmentId])
}

model Appointment {
  // ... existing fields ...
  feedbacks PatientFeedback[]
}
```

**Migration steps.**

1. Add the column (nullable) so existing rows stay valid.
2. Backfill by looking up the most recent appointment per `(patientId,
   submittedAt)` pair.
3. Update the `/feedback` write path to accept and store `appointmentId`.
4. Update `patientCsatAiFlow` in `kpi-metrics.ts` to join on
   `appointmentId` where the appointment's notes contain the triage marker.

## 2. `FrontDeskCall` model (for Â§3.9 call-volume KPI)

**Why.** MedCore has no phone-system integration today. The PRD KPI "front-desk
call volume for triage questions" is inherently external â€” the numbers live in
Twilio / Exotel / Ozonetel CDRs, not in our DB.

**Proposed change.**

```prisma
model FrontDeskCall {
  id          String   @id @default(uuid())
  tenantId    String?
  tenant      Tenant?  @relation(fields: [tenantId], references: [id], onDelete: SetNull)
  calledAt    DateTime
  durationSec Int
  fromPhone   String
  toPhone     String?
  category    String?   // "TRIAGE" | "BILLING" | "APPT_CHANGE" | "OTHER"
  disposition String?   // "RESOLVED" | "ABANDONED" | "TRANSFERRED"
  providerId  String?   // external CDR id (Twilio SID, etc.)
  createdAt   DateTime @default(now())

  @@index([tenantId])
  @@index([calledAt])
  @@index([category])
  @@map("front_desk_calls")
}
```

A lightweight webhook receiver (`POST /api/v1/telephony/cdr`) would insert
rows from the telephony provider. Until that's wired, the KPI stays
`unavailable: true` with a pointer to this file.

## 3. `AIScribeSession.doctorNps` / `doctorRating` (for Â§4.9 doctor NPS)

**Why.** `PatientFeedback.nps` captures the patient side. There is no
doctor-side rating of the scribe itself, so the PRD KPI "Doctor NPS for the
scribe (>40)" cannot be computed.

**Proposed change.**

```prisma
model AIScribeSession {
  // ... existing fields ...
  doctorNps       Int?     // -100..+100 NPS, set at sign-off
  doctorRatedAt   DateTime?
  doctorRatingComment String?
}
```

Plumb a single star + NPS prompt into the post-sign-off toast in
`apps/web/src/app/dashboard/scribe/page.tsx`, then compute:

```
(promoters - detractors) / total
  where promoters = doctorNps >= 9, detractors = doctorNps <= 6
```

## 4. `MedicationIncident` model (for Â§4.9 med-error KPI)

**Why.** Comparing scribe vs manual medication-error rates requires incident
tracking. Today MedCore's closest approximation is
`MedReconciliation.discrepancies`, but those are mismatches between admission
and discharge lists, not true errors. Returning `unavailable: true` is the
honest answer until an incident-reporting workflow is in place.

**Proposed change.**

```prisma
model MedicationIncident {
  id             String   @id @default(uuid())
  tenantId       String?
  tenant         Tenant?  @relation(fields: [tenantId], references: [id], onDelete: SetNull)
  patientId      String
  patient        Patient  @relation(fields: [patientId], references: [id])
  prescriptionId String?
  prescription   Prescription? @relation(fields: [prescriptionId], references: [id])
  reportedByUserId String
  reportedByUser   User   @relation(fields: [reportedByUserId], references: [id])
  reportedAt     DateTime @default(now())
  severity       String   // "NEAR_MISS" | "MINOR" | "MODERATE" | "SEVERE"
  scribeSessionId String?
  scribeSession   AIScribeSession? @relation(fields: [scribeSessionId], references: [id])
  narrative      String   @db.Text
  resolvedAt     DateTime?

  @@index([tenantId])
  @@index([reportedAt])
  @@index([scribeSessionId])
  @@map("medication_incidents")
}
```

The `scribeSessionId` link lets `medicationErrorRateComparison` cleanly split
the numerator by scribe vs non-scribe.
