## 1. Auto-deploy fixed (was failing on `Permission denied`)

**Root cause** found by inspecting the `dd13bea` deploy run logs: the
`Deploy to dev server` job's SSH session ran fine, but the script invocation
itself failed:

```
bash: line 1: /home/.../medcore/scripts/deploy.sh: Permission denied
exit code 126
```

`scripts/deploy.sh` (and every other `.sh` file in `scripts/`) was committed
to git as `100644` — no executable bit. A fresh clone on the dev server (or
on any Linux box) lacks the bit, so `ssh ... '<path>/deploy.sh'` exits 126
the first time CI tries to deploy non-interactively. Up until auto-deploy,
the manual SSH workflow had been invoked via `bash scripts/deploy.sh` so
the bit never mattered.

**Fix shipped in this commit:**
1. `.github/workflows/test.yml` now invokes via `bash <path>` so the file
   mode is irrelevant.
2. `git update-index --chmod=+x` applied to all 11 `scripts/*.sh` so future
   fresh clones get the bit set.
3. `scripts/deploy.sh` header rewritten to declare GHA as the primary
   invoker and the manual fallback as the exception path.
4. `docs/DEPLOY.md` restructured: GHA-first canonical path + manual
   fallback runbook + new troubleshooting subsection for auto-deploy
   failure modes.

**Verify the fix landed:**

```bash

  --json headSha,conclusion,createdAt \
  --jq '.[] | "\(.headSha[:8]) \(.conclusion) \(.createdAt[:16])"'
```

Expect the latest run on `main` to show `success` on the `Deploy to dev
server` job and the `Smoke-check public endpoints` step. Then:
